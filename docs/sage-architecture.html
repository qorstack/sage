AGENTS.md เป็น source of truth — integrations เป็นแค่ pointer ที่บอก AI ให้อ่าน AGENTS.md; knowledge เติบโตใน agents/sage/ ของ repo ที่ใช้งาน

Sage ประกอบด้วย 4 layer ที่ทำงานต่อกัน — แต่ละ layer มีหน้าที่ชัดเจน และไม่มีอะไรที่ซ้อนทับกัน: AGENTS.md คือ protocol, agents/sage/ คือ knowledge ของทีม, integrations/ คือ adapter บางๆ ที่เชื่อม AI tool กับ protocol, และ docs/ คือ output ที่มนุษย์อ่านได้ ทั้ง 4 ชิ้นนี้ต่อกันเป็น feedback loop เดียว

AGENTS.md — หัวใจของ Protocol

Responsibility
Full cognition pipeline — roles, pipeline §1–4, knowledge format, enforcement, output blocks
Tech
Single Markdown file · ไม่มี server ไม่มี build — อ่านได้ทุก AI tool
Exposes
Protocol ที่ AI follow · role loading hook · summary block format · enforcement levels
Consumes
ไม่มี dependency — คือ root ของทุกอย่าง

AGENTS.md คือ Sage ทั้งหมด ไม่มี server, ไม่มี Python, ไม่มี build step — AI agent ใดก็ตามที่อ่านไฟล์นี้ก่อนทำงานจะ follow the protocol อัตโนมัติ สิ่งที่อยู่ใน AGENTS.md แบ่งเป็นสองช่วง: ก่อนเขียนโค้ด และหลังเขียนโค้ด

ก่อนเขียนโค้ด — Pipeline §1–4

  1. Become the role — เลือก senior lens ที่เหมาะสม (dev, architect, qa, designer, …) แล้วเปิด agents/sage/roles/role-<lens>.md เพื่อรับ persona ก่อนเริ่มคิด
  2. Read the knowledge — เปิด agents/sage/<domain>/rules.md และ decisions/*.md ที่เกี่ยวข้อง เพื่อรู้ว่าทีมตัดสินใจอะไรไปแล้วบ้าง
  3. Find reusable assets — อ่าน source file ของ service/util/component ที่ rules ชี้ไว้ก่อนเขียนโค้ดที่ใช้มัน ไม่สร้างซ้ำสิ่งที่มีอยู่แล้ว
  4. Assess & plan — ประเมิน verdict (proceed / warn / ask / reject) แบ่งงานเป็น [parallel] / [sequential] และ assign effort tier ก่อน commit

หลังเขียนโค้ด — mandatory capture

  1. เขียน pattern ที่เรียนรู้ได้กลับเข้า agents/sage/<domain>/decisions/<slug>.md
  2. Output summary block บังคับ: role · domain · risk · done · decisions · validated · knowledge
📌
Single source of truth. ห้าม fork wording ของ protocol เข้าไปใน integration files — integration files แค่ pointer มาที่ AGENTS.md เท่านั้น เพื่อให้ทุก tool อ่าน protocol เดียวกันเสมอ

Protocol อ่านได้อย่างเดียวไม่พอ — มันต้องมี knowledge ของ codebase นั้นๆ ด้วย นั่นคือหน้าที่ของ layer ถัดไป

agents/sage/ — Knowledge Store

Responsibility
Team knowledge store — rules, decisions, skills, role personas ที่เติบโตได้
Tech
Markdown + YAML frontmatter · 1 idea ต่อ 1 ไฟล์ · git-versioned
Exposes
Domain rules · past decisions · skill playbooks · role personas — AI อ่านก่อนทุก task
Consumes
AGENTS.md — กำหนด format ของทุก entry และเวลาที่ต้องอ่าน

Knowledge store คือสิ่งที่ทำให้ Sage แตกต่างจาก system prompt ทั่วไป — มันเติบโตได้ ทั้ง human เขียนเอง และ AI เขียนให้หลังทุก task ทุก entry เป็น YAML frontmatter + Markdown body หนึ่งไฟล์ต่อหนึ่ง idea ทำให้ diff ใน git อ่านง่าย และ review ได้ทีละ decision

Directory layout

agents/sage/
  index.md                        # table of contents
  roles/
    role-dev.md                   # Senior Developer persona
    role-architect.md             # created on first use
    role-writer.md                # created by sage-docs
  <domain>/                       # e.g. billing, auth, frontend
    rules.md                      # standing rules for this domain
    decisions/
      <slug>.md                   # one team decision per file
    skills/
      <slug>.md                   # best practice / how-to playbook
  docs/
    docs-style-template.md        # CSS + JS template — inlined by sage-docs

แต่ละ entry มี frontmatter fields ที่ควบคุมพฤติกรรมของ AI: status: approved ได้รับน้ำหนักมากกว่า proposed และ enforcement: block หมายความว่า AI ต้อง reject task ถ้าจะฝ่าฝืน rule นั้น — ไม่ใช่แค่เตือน

FieldValuesความหมาย
typeteam_decision · convention · skill · business_contextประเภทของ entry — กำหนดว่า AI จะใช้มันยังไง
statusproposed · approved · deprecatedAI อ่านทุก entry แต่ approved ได้น้ำหนักสูงสุด ส่วน deprecated จะถูก ignore
enforcementblock · warn · adviseblock = reject task, warn = ดำเนินการต่อพร้อมเตือน, advise = แนะนำเท่านั้น
sourcehuman · aiบันทึกว่าใครเขียน — human entries ได้ trust level สูงกว่า
applies_todomain name หรือ file globAI อ่าน rule เฉพาะเมื่อกำลังทำงานกับ file ที่ตรงกับ pattern นี้

Knowledge store อยู่ใน repo ของทีม ไม่ใช่ใน Sage repo — ทำให้ทีม เป็นเจ้าของความรู้ของตัวเอง แต่ AI tool ที่ใช้งานต้องรู้จัก Sage protocol ก่อน นั่นคือหน้าที่ของ integrations

Integrations เป็นเพียง routing layer — แต่ละ adapter บอก AI tool ว่า "อ่านและทำตาม AGENTS.md" ไม่มี logic ของตัวเอง ไม่ duplicate protocol ทีมทำแค่ cp -r integrations/<tool> . ครั้งเดียวแล้วทุก AI session จะเริ่มด้วย Sage pipeline อัตโนมัติ

เมื่อ integrations พาให้ AI ทำงานตาม protocol สิ่งที่ AI ผลิตออกมาเพื่อคนอ่าน ไปอยู่ใน layer สุดท้าย

Responsibility
Self-contained HTML docs ที่ทีมอ่านได้ — flow diagrams, API specs, architecture
Tech
Static HTML · CSS inlined · inline SVG diagrams · ไม่มี external dependency
Exposes

ทุก HTML file ใน docs/ เป็น single file สมบูรณ์ — CSS inline, JS inline, SVG inline ไม่มีการ fetch ใดๆ เมื่อเปิดในบราวเซอร์ diagram มี zoom/pan ในตัว และทุก node ที่มีส่วนอธิบายด้านล่างจะ clickable ให้ jump ไปยัง section นั้นได้ทันที

ทั้ง 4 layer ทำงานเป็น feedback loop เดียว — ไม่ใช่ pipeline ที่ทำครั้งเดียวจบ

landing/index.html เป็นหน้าแรกที่คนนอกเห็น — แยกอิสระจาก AGENTS.md และ agents/sage/ ทั้งหมด แก้ไขได้โดยตรงโดยไม่กระทบ protocol ทั้งนี้ wording ใน landing ต้องไม่ duplicate protocol — แค่อธิบายว่า Sage คืออะไรและ install ยังไง

ทั้งสามส่วนนี้ไม่ได้ทำงานเป็น step แยกกัน — มันเป็น lifecycle เดียว ที่วนซ้ำทุก AI session ดังนี้

Data Flow — ทุกอย่างทำงานร่วมกันอย่างไร

ครั้งแรกที่ทีมใช้ Sage มีสองขั้นตอน one-time setup จากนั้นทุก AI session หลังจากนั้นจะวน loop เดิมอัตโนมัติโดยที่ทีมไม่ต้องทำอะไรเพิ่ม

  • 1
    Install integration (one-time)
    cp -r integrations/.claude . — copy adapter files เข้า repo ของทีม Codex และ Gemini CLI ไม่ต้อง copy เพราะอ่าน AGENTS.md ได้โดยตรง
  • 2
    Bootstrap knowledge (one-time)
    รัน /sage-learning ครั้งแรก — AI scan codebase แล้วเขียน rules, decisions, skills เข้า agents/sage/ ใน repo ของทีม ตั้งแต่นี้ไป knowledge store จะเติบโตเองทุก session
  • 3
    AI อ่าน AGENTS.md ก่อนทุก task
    Adapter บอก AI ให้อ่าน AGENTS.md → AI run pipeline §1–4 โดยอัตโนมัติ: pick role → read domain knowledge → find reusable assets → assess & plan ก่อนเขียน code แม้แต่บรรทัดเดียว
  • 4
    AI เขียนโค้ด + capture knowledge
    หลังเขียนโค้ด AI เขียน pattern ที่เรียนรู้ได้กลับเข้า agents/sage/<domain>/decisions/ ทุก run — ความรู้ของทีมสะสมขึ้นเรื่อยๆ และ AI session ถัดไปจะฉลาดกว่าเดิม
  • 5
    Generate human-readable docs (on demand)
    รัน /sage-docs → AI วิเคราะห์ source material → classify doc type → generate inline SVG diagram → write docs/<slug>.html ด้วย CSS inlined สามารถ update ไฟล์เดิมได้โดย Sage จะ merge content ที่ยังถูกต้องและ update เฉพาะส่วนที่เปลี่ยน
